# 서버 액티브 ETF 데이터 수집 분석 리포트

## 1. 수집 아키텍처 개요

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│  crontab    │────▶│  start.sh        │────▶│  app.py     │
│  (5분 주기) │     │  (watchdog)      │     │  (Flask)    │
└─────────────┘     └──────────────────┘     └──────┬──────┘
                                                     │
                                              APScheduler
                                              (매일 20:00)
                                                     │
                                                     ▼
                                             ┌──────────────┐
                                             │ 네이버 증권   │
                                             │ 크롤링 (25개) │
                                             └──────┬───────┘
                                                     │
                                                     ▼
                                             ┌──────────────┐
                                             │ SQLite DB    │
                                             │ active_etf.db│
                                             └──────────────┘
```

## 2. 수집 메커니즘

### APScheduler (핵심 수집)
- **트리거**: `cron`, 월~금 **20:00** 자동 실행
- **함수**: `run_collection()` → `collect_all_etf_data()`
- **대상**: 25개 액티브 ETF (config.py의 ETF_LIST)
- **방식**: 네이버 증권 HTML 파싱 (BeautifulSoup4)
- **안전장치**: ETF 간 **1.5초 sleep**, 실패 시 해당 ETF만 스킵
- **중복 방지**: 직전 수집일과 데이터 동일하면 새 날짜로 저장하지 않음

### 수집 흐름
```
1. APScheduler 20:00 트리거
2. run_collection() 호출 (threading.Lock으로 중복 실행 방지)
3. collect_all_etf_data() 실행
   → 25개 ETF 순회
   → ETF별 네이버 증권 구성종목 페이지 크롤링
   → euc-kr/UTF-8 인코딩 처리
   → 1.5초 sleep
4. SQLite DB에 collect_date 기준 저장
5. 결과 로깅 (저장 N, 오류 N)
```

## 3. 프로세스 생존 보장 (Watchdog)

### start.sh 스크립트
```bash
# crontab이 주기적으로 실행
# curl로 헬스체크 → 응답 없으면 프로세스 재시작
```

### crontab 설정
| 주기 | 작업 | 설명 |
|------|------|------|
| */5분 | `start.sh` | **앱 생존 확인 + 자동 복구** |
| 매일 00:00 | logrotate | 로그 파일 로테이션 |

### Watchdog 동작 방식
- 주기적으로 헬스체크 시도
- 실패 시: 기존 프로세스 종료 → 앱 재시작
- 성공 시: 아무 작업 없음

## 4. 운영 안정성 분석

### 수집 이력 (DB 기준)
- **최초 수집일**: 2026-02-22
- **최신 수집일**: 2026-03-13
- **총 수집 횟수**: 15일 (영업일 기준)
- **일평균 레코드**: ~250건 (25 ETF × ~10개 종목)

### 장애 이력

**2026-02-22: 반복 재시작 루프**
- watchdog.log에서 확인: 앱이 5분마다 반복 재시작
- 원인 추정: 초기 배포 시 앱이 안정적으로 기동되지 못함 (포트 충돌, 의존성 문제 등)
- 해결: 2026-02-23 이후 안정화

**현재 상태**: 정상 운영 중
- app.log 기준 APScheduler 정상 등록 확인
- 20:00 수집 정상 실행 중

### 주말/공휴일 처리
- APScheduler `day_of_week="mon-fri"` 설정으로 **주말 자동 제외**
- 공휴일은 별도 처리 없음 (네이버에서 데이터 변동 없으면 중복방지 로직으로 자연스럽게 스킵)

## 5. 잠재 리스크

| 리스크 | 심각도 | 설명 |
|--------|--------|------|
| 앱 크래시 후 수집 누락 | 중 | watchdog이 5분 내 복구하지만, 20:00 정각에 죽어있으면 당일 수집 누락 |
| 네이버 HTML 구조 변경 | 높 | 파싱 로직 깨짐 → 전체 수집 실패 가능 |
| SQLite 동시 쓰기 | 낮 | 단일 프로세스이므로 현재는 문제없음 |
| 디스크 부족 | 낮 | DB 크기 작음, 로그 로테이션 설정됨 |
| 수동 수집과 자동 수집 충돌 | 낮 | `threading.Lock`으로 중복 실행 방지됨 |

### 수집 누락 시나리오
APScheduler는 **in-process** 스케줄러라서 앱이 20:00에 죽어있으면 수집이 안 됨. watchdog이 5분 이내 복구해도 이미 20:00이 지나면 당일 수집은 누락. 수동 수집(`/api/collect`)으로 보완 가능하나 자동 보정은 없음.

## 6. 개선 제안

1. **수집 실패 알림**: 수집 실패 시 Slack/텔레그램 알림 추가
2. **재시도 로직**: watchdog 복구 후 당일 수집 여부 확인 → 미수집이면 자동 실행
3. **수집 이력 테이블**: 수집 성공/실패/건수를 별도 테이블에 기록하여 모니터링
4. **공휴일 캘린더**: 한국 공휴일 자동 스킵 (불필요한 크롤링 방지)
